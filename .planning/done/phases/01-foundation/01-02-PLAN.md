<objective>
Create two Nginx config files (blue-green-blue.conf and blue-green-green.conf) with upstream blocks pointing to the blue (3001) and green (3002) ports respectively. Create scripts to deploy these configs to EC2 and switch between them via symlink + nginx -s reload for zero-downtime switching.
</objective>

<context>
From multi-container-service/nginx/nginx.conf (existing working pattern):
- Upstream block with `server 127.0.0.1:3000; keepalive 64;`
- server block on listen 80, server_name _
- proxy_pass http://api_backend;
- proxy_http_version 1.1
- Standard proxy headers: Host, X-Real-IP, X-Forwarded-For, X-Forwarded-Proto
- location /health proxies to http://api_backend/health

Key locked decisions:
- Blue port: 3001, Green port: 3002 (upstream targets)
- Host Nginx on port 80 serves blue-green traffic
- Symlink-based switching (not config rewriting)
- Two pre-written static config files
- Zero-downtime switch via nginx -s reload
- Do NOT touch the existing multi-container-service Nginx config (default)
- Nginx configs must NOT proxy /health to multi-container-service -- they proxy to blue/green API

Architecture note from multi-container-service: The existing multi-container-service Nginx config is at /etc/nginx/sites-available/default. The blue-green configs must use different filenames and a different sites-enabled entry (blue-green symlink) so they never conflict.
</context>

  <name>Task 1: Create nginx/blue-green-blue.conf and nginx/blue-green-green.conf</name>
  <files>nginx/blue-green-blue.conf, nginx/blue-green-green.conf</files>
  <action>
Create two Nginx config files locally in the repository:

**nginx/blue-green-blue.conf**:
```nginx
upstream blue_backend {
    server 127.0.0.1:3001;
    keepalive 64;
}

server {
    listen 80;
    server_name _;

    # Proxy to blue environment
    location / {
        proxy_pass http://blue_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;

        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    # Health check endpoint
    location /health {
        proxy_pass http://blue_backend/health;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
    }
}
```

**nginx/blue-green-green.conf**: Same structure but:
- upstream name: `green_backend`
- server: `127.0.0.1:3002`
- location / proxy_pass: `http://green_backend`
- location /health proxy_pass: `http://green_backend/health`

These are static files. The blue.conf always points to 3001, green.conf always points to 3002. Switching is done by flipping the symlink, not by editing these files.

<task type="auto">
  <name>Task 2: Create scripts/setup-nginx.sh and scripts/switch.sh</name>
  <files>scripts/setup-nginx.sh, scripts/switch.sh</files>
  <action>
Create two shell scripts:

**scripts/setup-nginx.sh**: Deploys Nginx configs to EC2
SSH to 13.236.205.122 using ec2-static-site-key and:
1. Create /etc/nginx/sites-available/ if it does not exist
2. Copy blue-green-blue.conf and blue-green-green.conf to /etc/nginx/sites-available/ using heredoc-over-SSH or scp
3. Remove any existing /etc/nginx/sites-enabled/blue-green symlink (cleanup old)
4. Create symlink: `ln -sf /etc/nginx/sites-available/blue-green-blue.conf /etc/nginx/sites-enabled/blue-green`
5. Validate: `nginx -t`
6. If valid, reload: `nginx -s reload`
7. Verify: `curl -s http://localhost/health` returns 200 (should proxy to blue by default)

**scripts/switch.sh**: Switches the active environment via symlink
Arguments: `$1` = "blue" or "green"

SSH to EC2 and:
1. Read argument: TARGET="${1}" (blue or green)
2. Validate: if not "blue" and not "green", exit 1 with error
3. Validate config file exists: `/etc/nginx/sites-available/blue-green-${TARGET}.conf`
4. Switch symlink: `ln -sf /etc/nginx/sites-available/blue-green-${TARGET}.conf /etc/nginx/sites-enabled/blue-green`
5. Validate: `nginx -t`
6. Reload: `nginx -s reload`
7. Echo: "Switched to {blue|green} environment"
8. Verify: `curl -s http://localhost/health` returns 200

Both scripts should:
- Accept SSH_KEY and HOST as environment variables or defaults
- Use `set -euo pipefail` for safety
- Handle errors gracefully with exit codes
- Print status messages

Example switch.sh usage: `./switch.sh blue` or `./switch.sh green`
</action>
